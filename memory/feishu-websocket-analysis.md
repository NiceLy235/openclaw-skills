# 飞书 WebSocket 工作机制说明

## 🎯 核心发现

**飞书 WebSocket 是事件驱动的按需连接**：
- 空闲时：WebSocket 断开，Health 显示 OFF ✅ **正常**
- 有消息：WebSocket 建立，Health 显示 ON ✅ **正常**
- 处理完：WebSocket 断开，Health 显示 OFF ✅ **正常**

## 📊 工作流程

```
空闲状态（大部分时间）
├─ WebSocket: 断开
├─ 网络连接: 0
├─ Health: OFF
└─ 状态: ✅ 正常

    ↓ 新消息到来

事件触发（短暂）
├─ WebSocket: 建立
├─ 网络连接: 1
├─ Health: ON
└─ 状态: ✅ 正常

    ↓ 处理消息

空闲状态（回到）
├─ WebSocket: 断开
├─ 网络连接: 0
├─ Health: OFF
└─ 状态: ✅ 正常
```

## 🔍 这解释了所有现象

| 现象 | 原因 | 状态 |
|------|------|------|
| Health 经常 OFF | 空闲时断开 | ✅ 正常 |
| 网络连接数 0 | 空闲时断开 | ✅ 正常 |
| 机器人能响应 | 事件时建立 | ✅ 正常 |
| 监控显示 ON | 检查配置 | ✅ 正常 |

## 📋 监控策略调整

### ❌ 旧策略（v2.0）
```bash
检查 Health = OFF → 认为断开 → 重启 Gateway ❌
```

### ✅ 新策略（v3.0）
```bash
检查配置 → 检查 API → 配置正确就不管 ✅
只在真正问题时报警 ✅
```

## 🎯 监控检查项

v3.0 监控脚本检查：
1. ✅ **配置状态**：AppID、AppSecret 是否正确
2. ✅ **Gateway 进程**：是否运行
3. ✅ **Channels 状态**：飞书通道是否启用
4. ✅ **飞书 API**：能否访问飞书服务器
5. ⚠️ **不再检查 Health**：因为 OFF 是正常的

## 🔄 连接模式对比

| 模式 | 特点 | Health 状态 | 适用场景 |
|------|------|------------|----------|
| **持久连接** | 一直保持 | ON（稳定） | 高频消息 |
| **事件驱动** | 按需建立 | OFF（正常） | 低频消息 |
| **Webhook** | 被动接收 | N/A | 生产环境 |

## ✅ 结论

**飞书 WebSocket 的"不稳定"不是问题，是特性！**

- ✅ 空闲时断开 = 节省资源
- ✅ 有消息时建立 = 快速响应
- ✅ 监控策略已更新 = 不再误判
